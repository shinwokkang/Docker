# Docker 완전 정복: 5-1강. 포트(Port)와 볼륨(Volume) 심층 탐구 및 실무 적용례 🔬

이전 시간에 비유를 통해 알아보았던 포트 매핑과 볼륨 매핑을, 이번에는 비유를 벗어나 **실제 IT 인프라와 네트워크 관점**에서 조금 더 기술적이고 깊이 있게 파헤쳐 보겠습니다. 또한 실무 현장에서는 이 기능들이 정확히 어떤 문제를 해결하기 위해 쓰이는지 구체적인 예시와 함께 알아봅시다.

## 1. 포트 매핑 심층 분석 (Port Mapping, `-p`)

### 🧠 기술적 원리: 왜 매핑이 필요한가?
Docker를 설치하면 내 컴퓨터(Host) 내부에 가상의 네트워크 스위치(일반적으로 `docker0`라는 브릿지 네트워크)가 생깁니다. 그리고 컨테이너를 실행할 때마다 이 가상 네트워크 안에서만 통신할 수 있는 **사설 IP(예: 172.17.0.2)**가 컨테이너에 할당됩니다.

문제는 이 172.x.x.x 대역의 IP는 오직 내 컴퓨터 안에서만 유효하다는 것입니다. 외부 인터넷이나 사내 망에 있는 다른 컴퓨터들은 이 IP의 존재 자체를 모릅니다. 외부에서 접속할 수 있는 유일한 주소는 내 컴퓨터(Host)가 원래 가지고 있던 진짜 IP뿐입니다.

따라서 외부의 트래픽을 컨테이너 내부로 꽂아주기 위해, Docker 호스트가 중간에서 **네트워크 주소 변환(NAT, Network Address Translation)**을 수행하게 만드는 것이 바로 `-p` 옵션입니다. 

* `docker run -p 8080:80 nginx`를 실행하면:
  - 호스트 컴퓨터의 방화벽(iptables 등) 규칙이 수정됩니다.
  - "이제부터 호스트의 8080 포트로 들어오는 모든 트래픽은, 내부망 172.17.0.x의 80 포트로 토스(포워딩)해라"라는 규칙이 세팅됩니다.

### 🏢 실무 활용 예시: 마이크로서비스 로컬 테스트 및 포트 충돌 방지
실무에서는 하나의 애플리케이션만 개발하지 않습니다. 보통 여러 개의 서비스(API 서버, 결제 서버, 인증 서버 등)를 동시에 띄워놓고 연동 테스트를 합니다.

만약 모든 서버가 스프링 부트(Spring Boot)로 개발되었다면, 기본적으로 모두 컨테이너 내부에서 **8080 포트**를 사용하려고 할 것입니다. 
하지만 내 컴퓨터(호스트)의 8080 포트는 단 하나뿐이므로, 일반적인 방식으로는 여러 서버를 띄울 수 없어 포트 충돌 에러가 납니다.

**💡 Docker 포트 매핑을 이용한 해결:**
- 인증 서버 실행: `docker run -p 8081:8080 auth-server`
- 결제 서버 실행: `docker run -p 8082:8080 payment-server`
- 유저 서버 실행: `docker 일run -p 8083:8080 user-server`

컨테이너들은 각각 독립된 공간에 있으므로 맘 편히 각자의 8080 포트를 사용하게 두고, 외부(개발자)에서 접속할 때는 8081, 8082, 8083으로 나누어 접속함으로써 **포트 충돌 문제를 완벽하게 해결**합니다.

## 2. 볼륨 매핑 심층 분석 (Volume Mapping / Bind Mount, `-v`)

### 🧠 기술적 원리: UnionFS와 데이터의 휘발성
Docker 이미지와 컨테이너는 겹겹이 쌓인 레이어 구조(Union File System)로 이루어져 있습니다. 이미지는 '읽기 전용(Read-Only)' 레이어이고, 컨테이너가 실행될 때 그 위에 '읽기/쓰기(Read-Write)' 가능한 얇은 레이어가 하나 덮어씌워집니다. 컨테이너 내부에서 발생하는 모든 파일 변경, DB 저장 등은 오직 이 최상단 레이어에만 기록됩니다.

하지만 컨테이너를 지우면(`docker rm`) 이 최상단 레이어도 함께 삭제됩니다. 즉, 데이터의 생명주기(Lifecycle)가 컨테이너의 생명주기와 완전히 묶여버리는 치명적인 문제가 발생합니다.

이를 해결하기 위해 **데이터의 생명주기를 컨테이너로부터 분리(Decoupling)**하는 기술이 볼륨 매핑입니다.
`-v` 옵션(Bind Mount 방식)을 사용하면, 호스트의 특정 폴더를 컨테이너 내부의 경로에 직접 덮어씌웁니다(마운트). 이 상태에서 컨테이너가 파일을 쓰면, 컨테이너의 휘발성 레이어가 아니라 **호스트 컴퓨터의 물리적인 하드 디스크에 직접 기록**됩니다.

### 🏢 실무 활용 예시 1: 데이터베이스(DB) 무중단 버전 업데이트
실무 운영 환경에서 MySQL 버전을 5.7에서 8.0으로 업그레이드해야 하는 상황을 가정해 봅시다. 일반적인 서버라면 백업, 재설치, 복원 등 험난한 과정을 거쳐야 합니다.

**💡 Docker 볼륨 매핑을 이용한 업그레이드:**
1. 기존 DB 운영 중: `docker run -v /data/mysql_prod:/var/lib/mysql mysql:5.7` 
   (모든 고객 데이터는 호스트의 `/data/mysql_prod` 폴더에 안전하게 쌓이고 있음)
2. 기존 DB 컨테이너 과감하게 삭제: `docker rm -f 기존DB컨테이너`
3. 새로운 버전의 DB 실행: `docker run -v /data/mysql_prod:/var/lib/mysql mysql:8.0`

새로운 8.0 버전의 컨테이너가 실행되면서, **이미 데이터가 가득 들어있는 기존 호스트 폴더를 그대로 물고 올라옵니다.** 데이터 유실 없이 순식간에 DB 버전 업그레이드가 완료됩니다.

### 🏢 실무 활용 예시 2: 로컬 개발 환경에서 소스 코드 실시간 동기화
프론트엔드(React, Vue 등)나 백엔드 개발을 할 때, 코드를 한 줄 수정할 때마다 새로운 이미지를 다시 빌드(build)하고 컨테이너를 껐다 켜야 한다면 생산성이 최악일 것입니다.

**💡 볼륨 매핑을 이용한 실시간 개발:**
```bash
docker run -v /Users/shinwookkang/my-project:/app node_dev_env
```
내 컴퓨터에 있는 소스 코드 폴더(`/Users/shinwookkang/my-project`)를 컨테이너 내부의 `/app` 폴더에 매핑합니다.
이렇게 하면 내 컴퓨터의 VSCode 같은 에디터에서 소스 코드를 수정하고 저장하는 순간, 컨테이너 내부의 파일도 0.1초 만에 동시에 바뀝니다. 컨테이너 내부에서는 변경을 감지하고 자동으로 서버를 재시작(Hot Reloading)하므로, 개발자는 **컨테이너 환경에서 실행하면서도 마치 내 컴퓨터에서 직접 개발하는 것처럼 빠르고 편하게 작업**할 수 있습니다.
