# Docker 완전 정복: 6강. 나만의 도커 이미지 만들기 (Dockerfile & Push) 🛠️

지금까지 우리는 남들이 만들어 놓은 도커 이미지(ubuntu, mysql, redis 등)를 다운받아서 쓰는 법을 배웠습니다. 이번 데모(강의의 3.2 파트)에서는 드디어 **"우리가 만든 애플리케이션을 도커 이미지로 직접 구워내는(Build) 과정"**을 다룹니다.

이 과정을 완벽하게 이해하기 위해, 강사님은 먼저 '수동으로 설치하는 과정'을 보여주고, 그것을 어떻게 `Dockerfile`이라는 마법의 주문서로 바꾸는지 설명합니다.

---

## 1. 수동 설치 vs Dockerfile 자동화 🔄

만약 아무것도 없는 빈 Ubuntu 컴퓨터에서 Python 기반의 웹 서버(Flask)를 띄우려면 어떻게 해야 할까요? 직접 터미널에 들어가서 아래의 명령어들을 하나하나 쳐야 합니다.

```bash
# 수동 설치 과정 (과거의 방식)
1. apt-get update (패키지 목록 업데이트)
2. apt-get install -y python3-flask (파이썬과 Flask 프레임워크 설치)
3. 내 컴퓨터에 있는 소스 코드(app.py)를 Ubuntu 컴퓨터로 복사해 넣기
4. export FLASK_APP=app.py (환경 변수 설정)
5. flask run --host=0.0.0.0 (웹 서버 실행!)
```

만약 컴퓨터 100대에 이 서버를 띄워야 한다면? 이 과정을 100번 반복해야 합니다. 끔찍하죠. 
그래서 이 **'수동 설치 과정'을 그대로 스크립트로 옮겨 적은 것이 바로 `Dockerfile`**입니다.

### 📝 Dockerfile 작성하기
프로젝트 폴더 안에 `Dockerfile` 이라는 이름의 빈 파일을 만들고 아래처럼 적어줍니다.

```dockerfile
# 1. 어떤 운영체제(베이스 이미지)에서 시작할 것인가?
FROM ubuntu

# 2. 운영체제 안에서 어떤 명령어들을 실행할 것인가? (수동 설치 1, 2번)
RUN apt-get update
RUN apt-get install -y python3-flask

# 3. 내 컴퓨터에 있는 소스 코드를 컨테이너 안의 어디로 복사할 것인가? (수동 설치 3번)
COPY app.py /opt/app.py

# 4. 환경 변수 설정 (수동 설치 4번)
ENV FLASK_APP=/opt/app.py

# 5. 컨테이너가 켜질 때 마지막으로 실행할 최종 명령어는 무엇인가? (수동 설치 5번)
ENTRYPOINT ["flask", "run", "--host=0.0.0.0"]
```
*(참고: 영상에서는 Python 2.7과 pip를 썼지만, 최신 실습 코드에 맞춰 Python 3 기반으로 더 깔끔하게 업데이트된 버전입니다.)*

## 2. 이미지 빌드(Build)와 푸시(Push) 🚀

주문서(Dockerfile) 작성이 끝났다면, 이제 이 주문서를 바탕으로 진짜 **도커 이미지(틀)**를 찍어낼 차례입니다.

### 1) 이미지 빌드하기 (`docker build`)
```bash
# 현재 폴더(.)에 있는 Dockerfile을 읽어서 'simple-webapp' 이라는 이름(-t)의 이미지를 구워라!
docker build -t simple-webapp .
```
이 명령어를 치면 Docker가 차례대로 `RUN` 명령어들을 실행하며 이미지를 만들어 냅니다.

### 2) 내 깃허브 계정처럼, 도커 허브 계정으로 이름표(Tag) 달기
만든 이미지를 전 세계 사람들이 쓸 수 있도록 Docker Hub 앱스토어에 올리고 싶다면, 내 계정 이름(예: `mmumshad` 또는 `shinwookkang`)을 앞에 붙여주어야 합니다.
```bash
# 내 계정 이름으로 다시 이름표를 붙여줍니다.
docker tag simple-webapp shinwookkang/simple-webapp
```

### 3) 로그인 후 업로드(`docker push`)
```bash
docker login # (아이디와 비밀번호 입력)
docker push shinwookkang/simple-webapp
```
축하합니다! 이제 전 세계 어디서나 `docker run -p 5000:5000 shinwookkang/simple-webapp` 명령어 한 줄이면 여러분이 만든 웹 서버가 똑같이 켜집니다!

## 💡 전공자 수준의 딥 다이브: 실무에서는 어떻게 활용될까? 

우리가 방금 작성한 몇 줄의 `Dockerfile`은 실무(IT 기업) 환경에서 혁명을 일으킨 핵심 기술입니다. 단순한 기능 명세를 넘어, 전공자 관점에서 이것이 어떤 아키텍처적 가치를 가지는지 4가지 핵심으로 풀어보겠습니다.

### ① Infrastructure as Code (IaC, 코드로 관리하는 인프라)
과거에는 서버 환경(OS 버전, 파이썬 버전, 라이브러리 목록)을 엑셀이나 위키(문서)에 기록했습니다. "김대리, 서버 2번 켤 때 파이썬 2.7 말고 3.0 깔아."라고 말로 소통했죠. 
하지만 이제는 **인프라 자체를 코드로 작성(`Dockerfile`)**합니다. 코드로 작성되므로 Git 버전 관리가 가능해지고, 누가 언제 어떤 라이브러리를 추가했는지(Commit 기록) 투명하게 추적할 수 있습니다. 

### ② 불변 인프라 (Immutable Infrastructure)
기존에는 서버 1대에 문제가 생기면 개발자가 서버에 원격 접속해서(SSH) 이것저것 고쳐보고 재시작했습니다. 시간이 지날수록 서버는 누더기가 됩니다.
하지만 Docker 기반 실무 환경에서는 **절대 돌아가고 있는 컨테이너 내부에 접속해서 패키지를 고치지 않습니다.** 
대신, 소스 코드나 Dockerfile을 수정한 뒤 **새로운 이미지를 아예 다시 빌드(V2)해서, 기존 컨테이너(V1)를 가차 없이 부수고 V2 컨테이너로 통째로 갈아 끼웁니다.** 이를 통해 어떤 서버든 항상 100% 동일하고 예측 가능한 상태를 유지합니다.

### ③ CI/CD (지속적 통합/배포) 자동화 파이프라인 연동
우리가 손으로 쳤던 `docker build`와 `docker push`는 실무에서 사람이 직접 치지 않습니다.
1. 개발자가 코드를 수정해서 GitHub에 푸시(Push)하면,
2. **Jenkins**나 **GitHub Actions** 같은 자동화 로봇이 이를 감지합니다.
3. 로봇이 자동으로 `Dockerfile`을 읽어 `docker build`를 하고, 테스트를 돌린 뒤, 회사 전용 도커 허브(Private Registry, 예: AWS ECR)에 `docker push`를 합니다.
4. 마지막으로 쿠버네티스(Kubernetes) 같은 운영 환경에 "새 이미지 나왔으니 기존 거 내리고 새로 띄워!" 하고 명령합니다.
이 모든 파이프라인의 **가장 핵심적인 톱니바퀴가 바로 오늘 배운 Dockerfile**입니다.

### ④ 레이어 캐싱(Layer Caching)을 통한 초고속 빌드
Dockerfile의 각 줄(명령어)은 도커에서 **하나의 레이어(Layer)**로 저장됩니다.
만약 여러분이 파이썬 코드를 한 줄 수정해서 다시 `docker build`를 한다면, 도커는 처음부터 `apt-get update`나 `python 설치`를 다시 하지 않습니다. 
도커는 "어? 1, 2번 명령어는 아까랑 똑같네?" 하고 **이미 저장해둔 캐시(Cache)를 0.1초 만에 재활용**합니다. 그리고 소스 코드가 복사되는 `COPY` 단계부터만 새로 작업합니다. 
이러한 캐싱 알고리즘 덕분에 하루에도 수십 번씩 1초 만에 새로운 이미지를 찍어내며 실시간으로 개발하고 배포할 수 있는 것입니다.
